Autonomous Memory Leak Remediation

End-to-End Demo Process Diagram

This HTML explains the complete demo lifecycle: environment setup in OpenShift, monitoring with Instana, webhook-based analysis by Bob AI Agent, automated source-code remediation through GitHub, CI/CD execution with Tekton, and redeployment through GitOps. The flow below represents the live process implemented by the repository.

Setup & cluster provisioning Runtime monitoring Alert detection & trigger AI analysis & code fix Build, deploy & verify

Demo Purpose

The demo starts with a Quarkus application that contains an intentional memory leak caused by a static in-memory collection. Instana detects abnormal memory growth, sends a webhook to Bob, Bob prepares a code fix in GitHub, and the delivery pipeline rebuilds and redeploys the corrected application on OpenShift.

Full Process Diagram

1. Setup & Provisioning
Setup

Load configuration

.env provides OpenShift, Instana, GitHub, webhook, and namespace values used by the setup flow.

Setup

Login & prepare cluster

scripts/setup-demo.sh logs into OpenShift, validates the external image registry route, and prepares the demo namespace plus the Instana agent namespace.

Setup

Create required secrets

Secrets are created for the Instana agent key, Bob webhook authentication, GitHub token, and Tekton GitHub webhook validation.

Setup

Build and push images

Bob and the Quarkus app are built locally, containerized with Podman, and pushed to the OpenShift registry using linux/amd64 images.

2. Deploy & Observe
Observe

Deploy Bob AI Agent

OpenShift applies the Bob service and deployment so the webhook listener is available inside the cluster.

Observe

Deploy Quarkus app

Kustomize resources deploy the intentionally vulnerable Quarkus application, service, and route.

Observe

Configure Tekton & GitOps

The pipeline, trigger template, event listener, external route, and ArgoCD application are applied.

Observe

Instana monitors runtime

The Instana agent captures JVM metrics, heap growth, and runtime health for the Quarkus pod.

3. Failure Detection
Detect

Leak is triggered

The demo or a manual call invokes POST /api/trigger-leak, which stores allocated byte arrays in a static collection and increases heap pressure.

Detect

Instana raises alert

Instana SaaS evaluates memory thresholds and leak symptoms, then generates a memory-related alert event.

Detect

Webhook sent to Bob

The alert payload is delivered to Bob’s webhook endpoint, authenticated with the shared secret header.

Detect

Bob validates context

Bob confirms the issue is memory related, normalizes the alert payload, and gathers supporting context.

4. Analysis & Fix Generation
Fix

Analyze service and code

Bob uses its analyzer and integrations to inspect the target repository and identify suspicious leak patterns.

Fix

Generate remediation

Bob prepares a source-code change that removes or corrects the leak pattern and validates the proposed fix.

Fix

Push branch / PR in GitHub

The fix is committed to GitHub and exposed through a pull request for visibility and merge automation.

Fix

GitHub event emitted

A GitHub webhook event reaches the external Tekton EventListener route to start the delivery process.

5. Build, Deploy & Verify
Deploy

Tekton creates PipelineRun

The EventListener and TriggerTemplate create a new PipelineRun for the corrected repository revision.

Deploy

Clone, test, and build image

Tekton fetches the repository, runs Maven stages, builds the container image, and pushes it to the registry.

Deploy

GitOps sync deploys fix

ArgoCD detects the desired state or updated artifact flow and reconciles the fixed version into OpenShift.

Deploy

Instana confirms recovery

The fixed application runs with improved behavior, memory pressure stabilizes, and the incident is considered resolved.

The flow above combines the setup automation from scripts/setup-demo.sh, the architectural description in README.md and docs/ARCHITECTURE.md, and the Tekton trigger pipeline implemented in the repository.

Operational Sequence

1
Bootstrap the environment

Run scripts/setup-demo.sh to create namespaces, secrets, images, deployments, pipeline resources, and GitOps configuration.

2
Generate the failure

Use the demo script or a manual request to trigger the memory leak endpoint and intentionally increase JVM memory usage.

3
Detect and route the incident

Instana monitors the pod, detects the leak behavior, and sends an authenticated webhook event to Bob.

4
Remediate through Git

Bob analyzes the code, generates a change, and creates a GitHub branch and pull request to carry the correction.

5
Rebuild and redeploy

Tekton executes the CI pipeline and OpenShift/GitOps deploy the corrected build so Instana can verify resolution.
